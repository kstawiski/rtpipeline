**Scientific Concern: InstanceNumber-based Sorting**

The function sorts slices by InstanceNumber (line 54):
```python
series.sort(key=lambda x: (x.instance_number is None, x.instance_number or 0))
```

**DICOM Standard Position on InstanceNumber:**

According to the DICOM standard and community best practices:
- **ImagePositionPatient (0020,0032)** is the MANDATORY and RELIABLE attribute for spatial ordering
- **InstanceNumber (0020,0013)** is UNRELIABLE - some vendors set all slices to "1" or omit it entirely
- The correct method is to project ImagePositionPatient onto the plane normal vector

**References:**
- DICOM PS3.3 C.7.6.2 Image Plane Module
- comp.protocols.dicom archives on slice sorting

**Impact Assessment:**

This sorting is used for:
1. **Indexing purposes** - Low impact (order doesn't affect grouping)
2. **File naming in copy_ct_series()** - HIGH IMPACT (see bug below)
3. **NIfTI conversion** - LOW IMPACT (dcm2niix does its own correct sorting internally using ImagePositionPatient)

**Verdict**: The InstanceNumber sorting doesn't affect NIfTI conversion quality since dcm2niix
handles this correctly. However, it could cause file naming collisions (see Bug #1 below).

### 2. COPY_CT_SERIES - **CRITICAL BUG IDENTIFIED** 🚨

**Bug Location**: Lines 66-71

```python
def copy_ct_series(series: List[CTInstance], dst_dir: Path, ...) -> None:
    ensure_dir(dst_dir)
    for inst in series:
        dst = dst_dir / f"CT_{inst.instance_number or 0}.dcm"  # <- BUG HERE
        if copy_manager is not None:
            copy_manager.copy_dicom(inst.path, dst, skip_if_exists=True)
        else:
            shutil.copy2(inst.path, dst)
```

**The Problem:**

If ALL InstanceNumber values are None (which some vendors do), then:
- ALL destination files are named `CT_0.dcm`
- Files either OVERWRITE each other (shutil.copy2 path) or
- Get SKIPPED (copy_manager with skip_if_exists=True)
- **Result: Only 1 slice preserved out of potentially hundreds**

**Severity: HIGH**
- Data loss scenario with certain vendor data
- Would produce incomplete NIfTI files
- Pipeline would appear to succeed but output would be unusable

**Root Cause:**
The code assumes InstanceNumber is always populated and unique, which violates DICOM reality.

**Recommended Fix:**

```python
def copy_ct_series(series: List[CTInstance], dst_dir: Path, ...) -> None:
    ensure_dir(dst_dir)
    for idx, inst in enumerate(series):
        # Use instance_number if valid, otherwise use enumeration index for uniqueness
        file_idx = inst.instance_number if inst.instance_number is not None else idx
        dst = dst_dir / f"CT_{file_idx:05d}.dcm"
        # ... rest of copy logic
```
